# План графической переработки Citizenship Valley (новая версия)

Этот документ — план следующего, существенно обновлённого поколения игры. Главная цель: **качественный скачок графики**, в первую очередь на игровом поле (локации, персонажи, анимации, фоновая жизнь), затем — здания (снаружи и внутри), меню/инвентарь/Character, мини-игры, и параллельно — более интересный и разнообразный обучающий процесс.

Предыдущий план (RPG-каркас, мини-игры, сюжет, Map Phase 1–5) закрыт и сохранён в `docs/GAMEPLAY_UPGRADE_OLD.md`. Всё, что там реализовано (стартовый поток, кастомизация, инвентарь, Progress Center, Character Panel, Story Beats, 7 мини-игр, Exam Simulation, vendors, curriculum tracking, QA-скрипты), **сохраняется** и является фундаментом. Новый план не ломает эти системы, а поднимает их визуальную планку.

## 1. Анализ текущей графики (что есть сейчас)

Технический аудит проведён по `game.js`, `styles.css`, `index.html`, `assets/`.

### 1.1 Технический рендер

- Canvas `1280×768`. Константы: `TILE = 48` (экранный), `LOGICAL_TILE = 32` (логический), `RENDER_SCALE = 1.5`. Весь мир рисуется в логических координатах, затем `ctx.scale(1.5)` + смещение камеры.
- Карты регионов — ASCII-сетки `30×19` символов (`#` стена, `~` вода, `=` доска/настил, `,`/`:` дорога/плаза, `T` дерево, `.` трава) → ~`960×608` логических пикселей на регион.
- **Вся графика мира — это вызовы `rect(x, y, w, h, color)`**: цветные прямоугольники, вручную собранные в псевдо-пиксель-арт. Нет ни одного спрайт-листа для мира.
- Конвейер кадра: `draw()` → `drawWorld()` → ground → path(plaza) → buildings → props → characters → ui-world; затем `drawScreenUi()`.
- Игровой цикл `loop()`: `movePlayer()` + `draw()` + `requestAnimationFrame`. **Нет delta-time, нет обновления NPC, нет фоновой анимации.** Анимация привязана к счётчику `player.step`.
- Ассеты: ленивый `new Image()`, рисование через `ctx.drawImage`, если `image.complete && image.naturalWidth`, иначе — fallback на примитивы (паттерн в `drawPropAsset` / `propAssetImage`). Уже используется для 7 предметов (`ITEM_ASSETS`) и 8 trigger-пропов (`PROP_ASSETS`).

### 1.2 Игровое поле — слабые места (главный приоритет улучшения)

- **Тайлы/террейн**: ограниченный набор, без переходов между типами (трава↔дорога↔вода), без кромок/углов, без вариативности, без глубины и теней. Берег, площадь, дороги — плоские.
- **Локации похожи между собой**: различаются в основном палитрой (`visual` региона) + несколько signposts/пропов/landmark. Силуэт и планировка узнаются слабо.
- **Главный герой** (`drawHero*`): уже есть 4 направления и 4 кадра ходьбы (bob/stride), цвет одежды, причёски, рюкзак, accent, held-tool. Но это rect-арт: мало кадров, нет idle-вариаций, нет бега, нет мягкой тени и сглаженной анимации.
- **NPC** (`drawPerson`): **все используют один шаблон тела**, отличаются только цветом пальто (по ключевому слову в имени) и крошечным аксессуаром. **NPC полностью статичны** — в `loop()` их никто не двигает. Нет фоновой жизни.
- **Нет атмосферы**: ни частиц, ни ветра/листвы, ни бликов воды сверх примитивных, ни света/времени суток, ни погодных акцентов.
- **Z-сортировка**: персонажи рисуются в фиксированном порядке (`npcs.forEach` затем игрок), без сортировки по `y` — возможны неверные перекрытия по глубине.

### 1.3 Здания — слабые места

- **Экстерьеры**: единственная функция `drawBuilding()` рисует все здания одинаково по структуре (крыша, дверь, два окна, вывеска); различия — цвет стены/крыши + крошечный орнамент по ключевому слову в `drawBuildingOrnaments` (колонны/книги/клумба/баннер). Назначение здания по силуэту не читается.
- **Интерьеры переиспользуются**: существует всего **4 шаблона интерьера** (`townHallInterior`, `libraryInterior`, `courtInterior`, `parkInterior`) с общей картой `studyInteriorMap`, и они назначаются всем 7 регионам. Например, «Printworks» в Modern Britain открывает обобщённый `libraryInterior`. `drawInteriorDecor()` переключается только по этим 4 id. Назначение конкретного здания внутри не отражается.

### 1.4 Меню / HUD / инвентарь / Character / мини-игры

- UI — DOM + CSS (overlay-панели, карточки, метры). Предметы используют PNG-иконки с CSS-fallback. Портрет героя собран из CSS-блоков (`hero-portrait-*`).
- Мини-игры — DOM-панели с тематическими layout-паттернами и пропами (сделано в прошлом плане), но без настоящего арта карточек/доски и без canvas-анимаций.
- В целом функционально и читаемо, но визуально это «прототип UI», а не цельный арт-стиль.

### 1.5 Обучение — слабые места (для раздела разнообразия)

- Региональные квесты генерируются `makeNpc`/`buildQuest` и сводятся к «спросить NPC → ответить на 1 вопрос с 3 вариантами». Тип взаимодействия почти везде одинаковый.
- `curriculum.js` содержит богатые корректные пояснения и метаданные (area/difficulty/statBoosts/miniGameRefs/examSkill), но в самих квестах используется мало: нет разнообразия форматов, нет интервального повторения, слабая обратная связь «почему».

### 1.6 Ассеты сейчас

- Заполнены: `assets/items/` (7 PNG), `assets/props/region/` (8 trigger PNG + SVG), частично `assets/ui/`, `assets/story/`.
- **Пусты** (только `.gitkeep`): `assets/characters/`, `assets/tiles/`, `assets/buildings/`. То есть спрайт-листов персонажей, тайлсетов и зданий ещё нет — это основной объём новой работы.

## 2. Художественное направление (art direction)

Цель — единый, тёплый, читаемый «cosy civic top-down» стиль (в духе современных indie-RPG), но лёгкий по весу и совместимый с текущим pixel-логическим гридом.

1. **Базовое разрешение арта**: логический тайл `32 px`, экран `×1.5`. Спрайты рисуем в логическом масштабе (1 арт-пиксель = 1 логический), чтобы совпадать с миром. Источники — PNG с прозрачностью; по возможности держим SVG-исходники рядом (как для trigger-пропов).
2. **Палитра**: фиксируем мастер-палитру (расширяем `docs/VISUAL_STYLE_GUIDE.md`) — единые тени/света, региональные подпалитры, цвета статусов (quest/mini-game/shop/story/exam).
3. **Обводка и читаемость**: интерактивные объекты (NPC, двери, trigger-пропы, study stations) получают единое мягкое контурное выделение и тень; неинтерактивный декор — без обводки.
4. **Глубина**: тени-овалы под персонажами/зданиями, лёгкий ambient occlusion у оснований, z-сортировка по `y`.
5. **Региональная идентичность через силуэт, а не только цвет**: каждый регион — узнаваемый landmark, тип мощения, тип зелени/воды, набор зданий-«видов».
6. **Производительность важнее эффектности**: спрайт-листы вместо десятков `rect`, выключаемые ambient-эффекты, бюджет ассетов (см. §10).

Технический инвариант стиля: **любой новый арт обязан иметь fallback** на текущий примитивный рендер (как `drawPropAsset`), чтобы игра не ломалась при отсутствии/недогрузке изображения и проходила существующие visual smoke.

### 3.1 Террейн и тайлсет

2. **Autotiling кромок**: по ASCII-карте определять соседей и подбирать краевые/угловые тайлы (трава→дорога, суша→вода с пеной, площадь→газон). Карты остаются источником истины — меняется только их визуализация.

5. **Вода**: улучшенная анимация (бегущая рябь/блики через спрайт-кадры или процедурно), береговая пена у `~`-кромок (особенно Participation Harbour).

### 3.3 NPC: распознаваемость + фоновая жизнь

Две задачи: (а) сделать NPC визуально разными по роли; (б) добавить движение и фоновые активности.

1. **Распознаваемость**: спрайты/силуэты по роли (vendor, official/судья, editor, officer, youth worker, campaigner, elder и т.д.), а не только цвет пальто. Маленькие ролевые атрибуты (мантия, бейдж, фартук, планшет).
2. **Фоновая жизнь — ходячие NPC (1–2 на локацию)**:
   - Ввести у части NPC поле маршрута (waypoints/patrol-зона) и обновлять их позицию в `loop()` через **новый delta-time апдейт** (`updateNpcs(dt)`), с учётом коллизий (`isBlocked`) и плавным разворотом.
   - Чтобы **не сломать инварианты маршрутов и взаимодействия**: ходячими делаем **выделенных ambient-walkers без квестов/мини-игр**; quest-giver, mini-game host, vendor и story-NPC остаются на якорных позициях (надёжность `E` и route QA). Патруль-зоны исключают двери, travel-gate, spawn, study stations.
   - Idle-активности для остальных: подметание, чтение доски, жест разговора, разворот к площади/игроку (направление взгляда).
3. **Фоновые реплики NPC (speech bubbles)** — *реализовано (2026-06-08)*:
   - Периодически над головой NPC всплывает маленький **пузырь реплики** с короткой фразой **по теме игры** (гражданство/регион), затем плавно исчезает. Цель — оживить локации и **ненавязчиво подкреплять обучение**.
   - **Контент по теме**: фразы берём из региональных тем/`curriculum.js` (key terms, «почему это важно», короткие civic-факты), а не случайный флавор. Подбор по `state.currentLocation` (например, в Rights & Law — про права/обязанности, в Democracy — про голосование/представительство). Контент **точный и age-appropriate**; пул фраз — редактируемая таблица (по аналогии с тем, как `curriculum.js` — источник пояснений).
   - **Кому**: в первую очередь ambient-walkers и idle-NPC; у quest/mini-game/story-NPC реплики не должны перекрывать quest-маркер и подсказку `E` (приоритет у функциональных маркеров).
   - **Тайминг**: редко и **со сдвигом фаз**, чтобы не «болтали» все разом и не шумело; не чаще 1 пузыря на NPC раз в N секунд, лимит одновременных пузырей на локацию. Реализация — таймеры в существующем delta-time апдейте.
   - **Рендер**: пузырь рисуется в слое персонажей с z-сортировкой, не перекрывает лицо/маркеры, обрезается по ширине, читается на desktop и mobile. **Reduced Motion**: либо полностью выключает авто-реплики, либо показывает статичный пузырь без анимации (как остальные ambient-эффекты).
   - **Инварианты**: чистый рендер + таймер; **без изменения save-схемы**, без влияния на коллизии/маршруты/достижимость; не мешает диалогам и выбору ответов (когда открыт диалог/вопрос — пузыри скрыты, как и движение walkers).
4. **Z-сортировка** персонажей по `y` в `drawCharacterLayer`, чтобы герой/NPC корректно перекрывались.
5. **Реакция на игрока**: ближний NPC поворачивается к герою; у NPC с квестом/мини-игрой — спокойный маркер над головой (уже есть базовые маркеры — перенести на арт).

Файлы: `game.js` (новый `updateNpcs`, `drawPerson`/`drawCharacterLayer`, новый `updateNpcChatter`/`drawNpcSpeechBubble` + редактируемый пул фраз по региону, данные NPC: добавить `role`, `patrol?`, `facing?`), `curriculum.js` (источник тем для реплик), `loop()` (delta-time), `assets/characters/`. 

**QA-инвариант (критично)**: расширить `scripts/qa-route-audit.js` / `scripts/validate-world.js` — проверять, что (1) патруль-зоны ходячих NPC не пересекают двери/гейты/spawn/study stations; (2) достижимость всех статических host-NPC, дверей, гейтов и trigger-пропов не изменилась; (3) ходячий NPC не может «запереть» проход. Прогнать `qa-regional-playthrough.mjs` и `qa-regional-quests-playthrough.mjs`.

## 13. Рекомендуемый порядок ближайшей разработки

1. Согласовать с пользователем приоритет и объём арта (один общий тайл/спрайт-стиль).
2. **G0 → G1 → G2 → G3** (игровое поле — главный приоритет: террейн, герой, NPC + фоновая жизнь).
3. **G4 → G5** (здания снаружи и внутри).
4. **G6 → G7** (меню/инвентарь/Character и мини-игры).
5. **G8** (обучение) — параллельно может идти с G6–G7.
6. **G9** (атмосфера/полировка) и, по явному запросу, deploy.

Критерий успеха новой версии: игрок видит **узнаваемые, живые локации** (разный террейн, анимированный герой, 1–2 ходячих NPC и фоновая жизнь), **разные здания снаружи и тематические интерьеры внутри**, более **богатые меню/инвентарь/мини-игры** и **разнообразный, понятный обучающий процесс** — при сохранении лёгкости, совместимости сейвов и стабильного прохождения «регион → NPC → активность → награда → прогресс».

Документ живой: обновлять по мере выпуска этапов G0–G9 и появления решений.
